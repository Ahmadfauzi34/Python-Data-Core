## 2024-05-18 - [O(N^2) Vector Similarity Bottleneck] **Learning:** The pairwise distance calculation (`np.mean(np.cos(a-b))`) in `FHRRTopologicalLayer._ensure_distance_matrix` using nested Python loops is a massive CPU bottleneck, taking ~17.9 seconds for just 500 tokens. **Action:** Refactored into highly vectorized matrix multiplication using trig identity `cos(a-b) = cos(a)cos(b) + sin(a)sin(b)`. Brought runtime to ~0.15s (>100x speedup). Always look for trig identities to map circular operations to standard BLAS routines when vectorizing HRR/FHRR models.
## 2024-05-18 - [O(1) Vocabulary Lookup] **Learning:** Extensive use of `list.index()` across the engine (for roles and tokens) caused hidden O(N) performance degradations, particularly when parsing large queries or iterating topologies. **Action:** Maintained shadow dictionaries `_token_name_to_idx` and `_role_name_to_idx` alongside the lists to drop retrieval latency from ~0.50s to ~0.003s in micro-benchmarks. Always back frequently accessed categorical lists with hash maps in core loop architectures.
## 2024-05-18 - [O(MxN) FHRR Decode Bottleneck] **Learning:** `FHRREngine.decode` used nested Python loops over all `M` roles and all `N` tokens to compute FHRR similarity scores `cos(struct - role - token)`. This became extremely slow as vocabularies grew (~13.7s for 5000 tokens/20 roles). **Action:** Vectorized decoding using array broadcasting for `unbind` and matrix multiplication using trig identity `cos(A-B) = cos(A)cos(B) + sin(A)sin(B)` to compute all scores simultaneously. Runtime plummeted to ~1.2s (>10x speedup).
## 2024-05-18 - [FHRR Attention Vectorization] **Learning:** Loop-based algorithms mapping sequence lengths and attention heads over FHRR operations cause massive Python overhead. When dealing with phase continuous states (modulo 2π), avoid scalar interpolations. **Action:** Refactored attention components to use complex-plane vectorization (`np.angle(np.sum(np.exp(1j * bind_res)))`) to perform highly efficient, C-speed FHRR bundling across multidimensional arrays without a single Python `for` loop.
## 2024-05-18 - [FHRR Zero-Alloc Encoding] **Learning:** `FHRREngine.encode` used intermediate list buffers and array copies to compute structure bundles, leading to high Garbage Collection (GC) overhead during mass NLP ingests. **Action:** Refactored encoding to use "Zero-Allocation Phase FHRR". Directly accumulated bindings into a pre-allocated complex buffer (`self._cbuf += np.exp(1j * (R + T))`) eliminating list allocations and avoiding redundant modulo operations, dropping overhead by ~30%.
## 2024-05-18 - [Cognitive Prioritization] **Learning:** In cognitive architectures relying on multiple memory subsystems, checking deterministic databases (Knowledge Graph) first overrides probabilistic reasoning. **Action:** Refactored `query_api.py` to process queries in a human-like priority: Semantic Vector Inference -> Short-term Episodic Recall -> Long-term Knowledge Graph Rote Lookup.
## 2024-05-18 - [Unstructured Text Ingestion] **Learning:** Unstructured data ingestion requires a pipeline connecting chunking, entity extraction, OOTV handling, and episodic encoding. **Action:** Built a blueprint `TextIngestorBlueprint` to establish the interface logic between dependency parsing (currently a stub) and the highly optimized `FHRREngine` Zero-Alloc encoding loop.
## 2024-05-18 - [Multi-file Data Loading] **Learning:** Hardcoding application states to massive single configuration files (like a monolithic YAML) creates maintenance bottlenecks. **Action:** Upgraded `loader.py` to support directory resolution. It now recursively iterates through multiple YAML files within a folder and aggregates their states using a custom `_deep_merge` algorithm. This allows developers to cleanly split `vocab.yaml`, `observations.yaml`, etc., without altering the downstream load logic.

## 2024-05-18 - [Vectorization of KNN Masking in Spectral Affinity] **Learning:** When building an affinity matrix with a k-nearest neighbor constraint, applying a threshold per row via list comprehensions `[self.engine.sim(...) for j in range(n)]` inside a Python `for i in range(n)` is incredibly slow (~7s for N=500). **Action:** Compute the full matrix at once `(C @ C.T + S @ S.T) / dim`, find the thresholds simultaneously via `np.partition(sim_mat, -k, axis=1)[:, -k]`, and apply transformations via boolean masking `mask = sim_mat >= thresh[:, None]`. Brought runtime down to ~0.16s.

## 2024-05-18 - [Vectorization of Discovery K-Means] **Learning:** Hand-written 1D clustering loops using `[np.dot(v,c) for c in centroids]` inside a loop of length N are extremely inefficient. Repeated use of `np.pad` in every iteration exacerbates GC overhead. **Action:** Refactored `SelfSupervisedDiscovery.induce_roles` by pre-padding and normalizing all positional profiles into a single `(N, max_len)` matrix `X`. The clustering loop now relies purely on `X @ centroids_norm.T` and boolean mask updates `X[labels == k].mean(axis=0)`. Always prioritize matrix shape-alignment upfront over inline scalar padding.
